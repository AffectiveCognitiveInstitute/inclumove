    The following gives an overview of coding style and naming conventions. As a general rule of thumb, this project follows Unity's own coding style.

    Useful Information

    Here are some general things that are worth mentioning:

    Zenject

    As this project heavily makes use of the Zenject Framework it is highly recommended to read through the documentation on the GitHub page, which you can find here. This explains what the framework is for and how to use it. If you cannot find an answer, feel free to ask James.

    Namespaces

    All classes and structs should be in an appropriate namespace starting with Aci.

    Example:

    namespace Aci.Unity.UI
{
    public class Foo
    {

    }
}

    You needn't create a namespace for every folder you create either. Otherwise the namespaces will get extremely long.

    Classes / Structs / Enums

    Classes and Structs use the UpperCamelCase scheme:

    public class Foo
{

}

public struct Bar
{

}

public enum MyEnum
{
    None,
    TypeOne,
    TypeTwo
}

    This applies to all types of access modifiers (public - private - internal - protected) and abstract classes. Enum values also use the UpperCamelCase naming scheme.

    Interfaces

    Interfaces should start with the letter 'I' and use the following scheme: IUpperCamelCase. Example:

    public interface IConvertible
{
    void Convert();
}

    Properties / Events

    All properties and events use a lowerCamelCase naming scheme, regardless of access modifier.

    Example:

    public class Foo
{
    public event PropertyChangedDelegate propertyChanged;

    public GameObject gameObject { get; set;}
}

    Methods

    All methods, regardless of modifier, use the UpperCamelCase naming scheme.

    Example:

    public class Foo
{
    public void DoSomethingPublic()
    {

    }

    private void DoSomethingPrivate()
    {

    }

    private abstract void DoSomethingAbstract()
    {

    }

    private virtual void DoSomethingVirtual()
    {

    }

    public static void DoSomethingStatic()
    {

    }
}

    Variables

    Public / Internal

    Public and internal variable members of a struct or class follow the lowerCamelCase naming scheme.

    Example:

    public class MyClass
{
    public int myInteger;
    internal int anotherInteger;
}

    Private / Protected

    Private and protected variable members of a struct or class follow the following naming convention: m_UpperCamelCase.

    Example:

    public class MyClass
{
    private int m_MyInteger;
    protected int m_AnotherInteger;
}

    Local

    Local variables use a lowerCamelCase naming scheme.

    Example:

    private void Swap(int a, int b)
{
    int tempVariable = b;
    b = a;
    a = b;
}

    Consts

    Constants follow the UpperCamelCase naming scheme.

    Example:

    public class MyClass
{
    public const string Default = "A random string value";
}

    Static

    Private static members use the following naming scheme: s_MyStaticMember:

    Example:

    private static readonly int s_RandomValue;

    Public static variables use the same naming convention as public member variables.

    Order

    Though not strict, the order within a class, struct or interfaces goes as follows (following SA1201 and SA1203 convention):

    • Constant Fields
    • Fields
    • Constructors
    • Finalizers
    • Delegates
    • Events
    • Enums
    • Interfaces (interface implementations)
    • Properties
    • Indexers
    • Methods
    • Structs
    • Classes

    With each group order by access:

    • public
    • internal
    • protected internal
    • protected
    • private
